## 1. Exact-Match School Scope Violations in Reports (Impact: 5/5)
**Status**: Unresolved / Architectural Defect
**Code Refs**:
- `ifitwala_ed/students/report/case_entries_activity_log/case_entries_activity_log.py`
- `ifitwala_ed/accounting/report/trial_balance/trial_balance.py`
- `ifitwala_ed/accounting/report/aged_receivables/aged_receivables.py`
- `ifitwala_ed/accounting/report/student_attribution/student_attribution.py`

**The Issue**:
The `nested_scope_contract.md` establishes a rigid rule: "Requested school scope must always be intersected with the caller's visible school subtree" and must use an `IN %(school_scope)s` filter to guarantee permissions and parity. An inspection of the actual Python implementations reveals that all four of these reports are still using exact-match exactness for filtering (e.g. `rc.school = %(school)s`, `gl.school = %(school)s`, and `sii.school = %(school)s`). This actively breaks sibling-isolation models and obscures descendant-school data from parent-school administrators.

**Concrete Steps to Take**:
1. Within each `execute` function, import `_resolve_school_scope` or `get_descendant_schools` from `ifitwala_ed.utilities.school_tree` (or tree utils).
2. Intersect the requested school with the authenticated caller's default/allowed subtree.
3. Rewrite the query constructions in these reports to replace `=` operators with `IN %(school_scope)s`.
4. Cast the array/list returned from the tree scope tool as a `tuple()` passing it into the DB parametrics to avoid SQL syntax errors on single entries.

## 2. In-Process State & Missing Redis Layer C Caching (Impact: 4/5)
**Status**: Needs Architecture Refactor
**Code Ref**: `ifitwala_ed/api/course_schedule.py`

**The Issue**:
According to the `caching_strategy.md`, "schedule resolution maps" represent a *Category A* cache target, meaning they should be cached aggressively using Frappe's Redis layer with explicit invalidations. Upon auditing `course_schedule.py`, the rotation and term caches:
```python
schedule_cache: Dict[Tuple[str, str], Optional[str]] = {}
rotation_cache: Dict[Tuple[str, str], Dict[str, int]] = {}
term_cache: Dict[str, dict] = {}
```
are implemented strictly as simple in-process Python dictionaries generated during the request lifecycle (*Layer A cache*). This provides no cross-worker reuse. At scale, this forces every application thread to redo DB queries to remap dates into rotation blocks rather than offloading that to a coordinated memory store.

**Concrete Steps to Take**:
1. Refactor schedule mappings to use `frappe.cache().get_value(key, generator=...)`. Make sure the namespaces integrate the parameters (e.g. `ifed:schedule_rot:{academic_year}:{school}`).
2. Write centralized invalidation rules (e.g., `invalidate_schedule_cache(...)`) in helper files.
3. Link the invalidation rules onto Document lifecycle hooks (`on_update`, `on_trash`) for models that inherently dictate the cache: `School Schedule Block`, `Student Group Schedule`, `Term`, etc.

## 3. False Positives Left in `anti_patterns.md` (Impact: 2/5)
**Status**: Documentation Drift / Code Has Already Evolved
**Code Refs**:
- `ifitwala_ed/api/attendance.py`
- `ifitwala_ed/schedule/doctype/program_enrollment/program_enrollment.py`
- `ifitwala_ed/api/course_schedule.py`
- `ifitwala_ed/schedule/doctype/course_enrollment_tool/test_course_enrollment_tool.py`

**The Issue**:
Points 5, 6, 9, and 10 inside `anti_patterns.md` denote heavy/looping N+1 anti-patterns that exist or are "observed from grep results". Extensive review of these exact codebase files reveals that the backend engineering team has *already resolved* these problems:
- `api/attendance.py` queries have been flattened into properly scaled grouped metric counters via `_compute_period_kpis` and bounds checking.
- `program_enrollment.py` line 648 explicitly uses a batched caching proxy `_term_meta_many` across the entire row slice specifically intended to strip out the latency of individual term queries in a loop.

- `test_course_enrollment_tool.py` was refactored and no longer loops heavy `frappe.get_doc` over arrays of student enrolments.

**Concrete Steps to Take**:
1. Update `ifitwala_ed/docs/anti_patterns.md` points 5, 6, 9, 10.
2. Edit their "Status" definitions from "Observed from grep results" to "Resolved in current code".
3. Add a line validating that the new behavior utilizes cached proxy methods (`_term_meta_many`), unified array parameters for `get_today_courses()`, and aggregate reads `_compute_period_kpis()`.

## 4. Strategic Pivot: Dual-Surface Student Hub / "Today Cockpit" (Impact: 5/5)
**Status**: Product MVP Gap / Opportunity
**Doc Refs**: `10_student_learning_hub_proposal.md`

**The Issue**:
A top-tier education ERP cannot rely on endless to-do feeds that fracture attention, especially for younger learners. Currently, the LMS and tasking interfaces are fragmented.

**Concrete Steps to Take**:
1. Evolve `ui-spa/src/pages/student/StudentHome.vue` into a dual-surface "Today Cockpit" by fetching a unified aggregation payload instead of running waterfall requests.
2. Implement a unified `My Work Board` pulling directly from `Task Delivery` (`task_delivery.py`) and `Task Outcome` doctypes for system-curated `Soon`/`Later` lanes, minimizing unbounded lists.
3. Keep the pure LMS reading-view contained within `ui-spa/src/pages/student/CourseDetail.vue` using ordered `Learning Unit` and `Lesson` schemas, while linking execution tasks via deep-linked focus mode overlays.

## 5. Unified Focus Routing Over Native ToDos (Impact: 5/5)
**Status**: Architecture Implementation Gap
**Doc Refs**: `05_focus_and_attention_system.md`

**The Issue**:
Relying on raw Frappe `ToDo` tables in the SPA breaks encapsulation and causes multi-tab workflow hunting. While Log Follow-ups and Inquiries are covered, critical ops like admissions, pastoral referalls, and grading are missing out.

**Concrete Steps to Take**:
1. Expand `ui-spa/src/overlays/FocusRouterOverlay.vue` to map new `action_type` strings to actionable Vue blocks:
   - `referral.triage.act.start` routing to `ReferralTriageAction.vue` (to handle `student_referral` records)
   - `applicant_review.assignment.decide` routing to `ApplicantReviewAssignmentAction.vue` (for `applicant_review_assignment`)
   - `leave_approval.act.decide` for `leave_application` records
   - `task_submission.grade.act` for rapid inline `task_submission` grading via the `Task Rubric`.
2. Do not introduce ToDo closing logic to the UI layer. Ensure `ifitwala_ed/api/focus.py` only serves as a read-only router facade, letting controllers like `assessment.py` actually finish workflows and automatically clear the Focus Item.

## 7. Transport Boundary & Contract Purity Enforcement (Impact: 4/5)
**Status**: Technical Debt / Stability Risk
**Doc Refs**: `01_spa_architecture_and_rules.md`

**The Issue**:
Type definitions frequently inherit `{ message: T }` Frappe transport wrappers, which bypasses IDE enforcement and causes silent JSON serialization issues.

**Concrete Steps to Take**:
1. Audit and amend TypeScript files inside `ui-spa/src/types/contracts/` to ensure they are fully decoupled from Frappe JSON transport envelopes.
2. Ensure `response.data.message` unwrapping occurs exclusively in `ui-spa/src/resources/frappe.ts` and never leaks into `useOverlayStack` or page hooks.
3. Refactor any service hooks under `ui-spa/src/lib/services/**/*.ts` to type-cast strictly against pure domain DTOs (like `FocusItemContext`) and handle validation failures before delivering payloads downward.

## 8. Decouple UX Feedback from Overlay Lifecycle (Impact: 4/5)
**Status**: Framework Non-Compliance (A+ Governance)
**Doc Refs**: `01_spa_architecture_and_rules.md`

**The Issue**:
Action overlays often attempt to orchestrate their own reload cycles, success toasts, or custom DOM events, causing "refresh storms" or blocking closures.

**Concrete Steps to Take**:
1. Remove all `.refetch()` calls and `toast` interactions embedded inside action-overlays beneath `ui-spa/src/overlays/`.
2. Limit `ui-spa/src/lib/services/**/*.ts` to dispatching exact global semantic signals like `uiSignals.emit(SIGNAL_TODAY_INVALIDATE)`.
3. Move full UX responsibility (fetching, toasting, layout resetting) to the "Refresh Owners" (e.g. `StaffHome.vue` or `CourseDetail.vue`), leveraging explicit `uiSignals.subscribe()` logic inside Vue `setup()` destructors.

**Concrete Steps to Take**:

2. Refactor raw Tailwind font-size utilities (`text-sm`, `font-semibold`) directly scattered inside `ui-spa/src/pages/**` components into canonical typography mappings like `.type-body` or `.type-meta`.
3. Delete any instances of `@apply` nested deep inside layout SFC `<style scoped>` chunks which breaks Tailwind v4 compiling determinism.

## 10. Portal Canonical Navigation & Scoped Form Theming (Impact: 3/5)
**Status**: Component Fragility
**Doc Refs**: `02_style_note.md`, `01_spa_architecture_and_rules.md`

**The Issue**:
Globally overridden form elements (`input`, `select`) corrupt Frappe-UI Teleport boundaries, and duplicate desktop/mobile drawer implementations induce routing bugs.

**Concrete Steps to Take**:
1. Enact a strict rule against global `<style>` overrides mapping to generic inputs inside `ui-spa/src/styles/app.css`. Sandbox all visual resets down to classes like `.ifit-filters input` or `.dialog-prompt select` to protect frappe-ui's internal `<Teleport>` DOM.
2. Restrict all responsive Student/Guardian navigations internally to `ui-spa/src/components/PortalLayout.vue` + `PortalSidebar.vue`.
3. Eliminate duplicate JS/media-query listeners scattered around shells by adopting single-truth state classes (`.portal-sidebar--expanded`, `.portal-sidebar--collapsed`) driven completely by `components.css`.
